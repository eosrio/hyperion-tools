Hyperion 4.5 · Architecture Vision

One stack serves every Antelope API shape
— each datum stored once.

A Hyperion operator already runs nodeos, Elasticsearch, and (since 4.0) MongoDB. Hyperion 4.5's goal: make that same infrastructure answer Hyperion v2 history, the Light API, AtomicAssets, and native chain v1 reads — without standing up a second database per app, and without storing any datum twice. State lives in the state tier; history lives in search; one SHiP feed keeps them current.

The one rule that wins on both storage and speed: history is never re-stored in the state tier, and state is never stored in history. Each query routes to the tier built for its access pattern.

The state tier, up close

One WormDB segment, many API namespaces — live from chain head, compacted in place

🗂 One .wseg file — namespaced tables

A single memory-mapped, columnar file. The table-ID space is partitioned so multiple API domains share one reader — add a namespace, not a database.

Forward store keyed by id + per-dimension inverted posting lists (owner / collection / schema / template / data-facet) + presorted orderings — hybrid roaring postings. Served embedded (no IPC), each datum once.

♻️ Live + self-compacting

1
Frozen base — immutable, mmap'd, read by binary search.
2
Freshness overlay — the SHiP feed applies mint/transfer/burn/setdata; reads merge base + overlay. The forward record is the sole arbiter, so stale postings are harmless.
3
Two-phase apply — base reads lock-free, a brief in-RAM commit → freshness lag in hundreds of µs, readers never stalled.
4
Compaction — fold base + overlay → a fresh segment in the background, then an atomic ArcSwap hot-swap. Overlay RAM reclaimed; readers keep serving.

✓ Proven

  • atomicdata decoder — byte-exact, 77/77 live schemas, 200/200 parity

→ Planned (the 4.5 target)
